In the last chapter you learned to download a stock's history. Now comes the obvious next question: *what do you do with hundreds of prices once you have them?* You compute things. The average. The daily moves. How wild the swings were. Whether two stocks dance together. And you want to do all of that on the whole column of prices at once - not by writing a slow loop that crawls through one number at a time.

That is exactly what **NumPy** is for. NumPy (say "num-pie") is the library that makes Python fast at number-crunching, and it's the engine underneath pandas, the indicators, and the backtester you'll use later. This chapter teaches just enough NumPy to turn a column of prices into the building blocks of every strategy: returns, cumulative growth, volatility, and simple statistical signals. We'll define every term as we go, and every number you see was computed against live market data.

:::info
You don't install NumPy separately - it comes along with the OpenAlgo SDK and pandas. At the top of each script we just write `import numpy as np`, and from then on every NumPy tool is reached as `np.something`. The `np` nickname is a near-universal convention; you'll see it in every codebase.
:::

## What is an array?

An **array** is NumPy's version of a list: an ordered row of numbers. So why not just use a Python list? Because an array is built for *math*. With a list, doubling every price means writing a loop. With an array, you write `prices * 2` and NumPy does it to all of them at once - faster, and in code that reads like the maths it represents.

Our prices arrive inside a DataFrame column (a pandas **Series**). One method, `.to_numpy()`, hands us the raw NumPy array. Once it's an array, handy summaries like `.mean()`, `.max()` and `.min()` are a single call away.

{{example: ch06/01_make_array.py | Build a price array from history | NSE}}

That `ndarray` in the output is NumPy's full name for an array ("n-dimensional array"). Ours is one-dimensional - a simple row. Notice how `prices.mean()`, `prices.max()` and `prices.min()` each crunched all 80 closes in one go.

:::tip
`prices[0]` is the first close, `prices[-1]` is the most recent - the same indexing you learned for lists in Chapter 2. The last element being `[-1]` is something you'll reach for constantly, because "the latest price" is usually what a strategy cares about.
:::

## Returns: the number that actually matters

Prices in rupees aren't directly comparable across stocks - a ₹10 move means something very different for a ₹100 stock than a ₹10,000 one. What traders really care about is the **return**: the *percentage* change. A **simple return** for one day is `(today's close − yesterday's close) / yesterday's close`.

Here's where arrays shine. To compute every daily return at once we use **slicing**: `prices[1:]` is "every price from the second onward" (today's), and `prices[:-1]` is "every price except the last" (yesterday's). Subtract one array from the other and NumPy lines them up element by element - no loop anywhere.

{{example: ch06/02_simple_returns.py | Simple daily returns, no loops | NSE}}

You always get *one fewer* return than you have prices - the very first day has no "day before" to compare against. From a single array of closes we now know the average daily move, the best day, and the worst day, each a one-line call.

## Log returns: the quant's preferred version

There's a second way to measure a return that professionals lean on: the **log return**, computed as `ln(today / yesterday)` - the natural logarithm of the price ratio. For small daily moves it's almost identical to the simple return, so why bother?

Because log returns **add up**. String together a week of simple returns and you have to *multiply* `(1+r)` terms to get the weekly figure; string together log returns and you just *sum* them. That additive property makes a lot of maths cleaner, which is why volatility and many models are built on log returns. This example proves the point: the sum of all the daily log returns exactly equals the log of the whole-period move.

{{example: ch06/03_log_returns.py | Log returns vs simple returns | NSE}}

See how the last-five simple and log returns are nearly the same numbers, yet the sum of every log return matches the log of the total move to the decimal. That clean additivity is the whole appeal.

:::key
**Simple return**: `(p1 - p0) / p0` - intuitive, what you'd quote a friend ("up 2% today"). **Log return**: `ln(p1 / p0)` - adds across time, the standard input for volatility and modelling. They agree closely for small moves and diverge for large ones. Knowing which is which saves real confusion later.
:::

## Vectorisation: the no-loop superpower

The technique behind everything above has a name: **vectorisation** - applying an operation to an entire array in one stroke instead of looping element by element. When you multiply, add, or compare an array, NumPy runs the operation across all elements in fast, compiled code. The bonus trick is **broadcasting**: combine an array with a single number and that number is automatically "stretched" to touch every element.

{{example: ch06/04_vectorised_math.py | Vectorised math with no loops | NSE}}

Three different vectorised ideas appear here. `prices / 83.0` broadcasts a scalar across the array (a rough rupee-to-dollar conversion). `prices - prices.mean()` measures each close's distance from the average. And `np.where(condition, 1, 0)` checks a condition on every element at once, returning 1 where it's true and 0 where it's false - perfect for counting, here counting the days that closed above average.

:::tip
Whenever you catch yourself about to write `for` over prices, pause - there is almost always a vectorised one-liner instead. It's not just shorter; on a year of intraday data the vectorised version can be hundreds of times faster. This habit is the single biggest performance win in all of market analysis.
:::

## Cumulative returns: what one rupee became

A list of daily returns is useful, but the question an investor really asks is: *if I'd held this from the start, what would I have now?* That's the **cumulative return** - the daily moves compounded together. Growth of one rupee is `(1 + r1) × (1 + r2) × (1 + r3) × ...`, and NumPy's `np.cumprod` (cumulative product) computes that running multiplication across the whole array at once.

{{example: ch06/05_cumulative_returns.py | Compound returns into a growth curve | NSE}}

A final value below 1.0 (here about 0.91) means one rupee *shrank* - the stock fell over the window, a roughly 8.5% loss. The `.max()` of the growth array tells you the highest the investment ever reached along the way, which is the seed of the **drawdown** idea we'll formalise when we backtest.

## Volatility: how jumpy is it?

**Volatility** measures how much a stock's returns bounce around - your read on its riskiness. The tool for it is the **standard deviation** (`std`), a statistic that captures the *typical spread* of numbers around their average. Small std means calm, steady returns; large std means wild swings.

Daily volatility is a small number, so it's convention to **annualise** it - scale it up to a yearly figure by multiplying by the square root of the number of trading days in a year (about 252). That `√252` factor is the standard way to stretch a daily figure to an annual one. Here we measure it on an MCX gold contract.

{{example: ch06/06_volatility.py | Annualised volatility with std | MCX}}

An annual volatility around 38% means gold's price has been swinging meaningfully - useful context before sizing a position. We feed `std` the **log returns** here, following the convention from earlier.

:::info
Volatility is symmetric - it counts big up days and big down days equally. It does *not* tell you direction, only *magnitude* of movement. A stock can be highly volatile while going nowhere. We'll use volatility for position sizing and risk in the strategy chapters; for now, just internalise that std = jumpiness.
:::

## Boolean masks: ask a question of every day

Here's one of NumPy's most useful tricks for traders. Compare an array to something and you don't get a single True/False - you get a **boolean mask**, a same-length array of `True`/`False`, one per element. Then you can use that mask to *filter*: `returns[mask]` keeps only the elements where the mask is `True`.

This is exactly how you hunt for specific days. Want every day that moved more than 2% in either direction? Build the mask `np.abs(returns) > 0.02` and filter with it. (`np.abs` is the absolute value - it strips the minus sign so a -3% day and a +3% day both count as "big".)

{{example: ch06/07_boolean_masks.py | Find big-move days with a mask | NSE}}

Two masks appear here: `np.abs(returns) > 0.02` isolates the big movers, and `returns > 0` separates up days from down days (with `~up_days` flipping the mask to get the down days). Counting `True`s is just `.sum()`, because Python treats `True` as 1 and `False` as 0. This filter-by-condition pattern is the literal foundation of every trading **signal** - we'll build entries and exits from masks just like these.

## Finding extremes: max, min, argmax

You often need not just the highest price but *when* it happened. NumPy gives you both: `.max()` and `.min()` return the *value*, while `.argmax()` and `.argmin()` return the *position* (the index) where that value sits. Feed that position back into the timestamps and you know the date of the peak.

{{example: ch06/08_min_max_argmax.py | Locate the high and low with argmax | NSE}}

The `arg` in `argmax` stands for "argument" - meaning the *position* that produces the maximum, not the maximum itself. Pairing `argmax` on prices with the DataFrame's date index is a clean way to answer "what was the high of the last six months, and which day made it?" - the basis of breakout and 52-week-high screens.

## Correlation: do two stocks move together?

When you hold more than one position, a vital question is whether they move *together*. **Correlation** answers it with a single number between -1 and +1: `+1` means two stocks' returns move in perfect lockstep, `0` means no relationship, and `-1` means they move exactly opposite. NumPy's `np.corrcoef` computes it.

A subtlety: `corrcoef` returns a small 2×2 **matrix** (a grid of numbers), and the figure we want - the correlation *between* the two - sits in the off-diagonal corner, `[0, 1]`. Here we compare two large private banks.

{{example: ch06/09_correlation.py | Correlation of two stocks' returns | NSE}}

A correlation of about 0.74 confirms what you'd guess - two big banks tend to rise and fall together, because the same news moves the whole sector. That matters for risk: holding both isn't true diversification, it's a double bet on banking. Correlation is the cornerstone of **pair trading** and portfolio risk, which we'll build on later.

:::warn
Correlation describes how two things *moved together in the past* - it is not a promise about the future, and it never implies one *causes* the other. Relationships shift, especially in a crisis when correlations famously lurch toward 1 and "diversified" books fall together. Treat any correlation as a recent observation, not a law.
:::

## Z-score: how stretched is the price?

Last tool, and a genuinely useful one. A **z-score** rewrites a price as *how many standard deviations it sits away from its average*. The formula is `(price − mean) / std`. A z-score of 0 means the price is exactly average; `+2` means it's unusually high (two std above the mean); `-2` means unusually low. It's a clean, scale-free way to ask "is this price stretched right now?"

{{example: ch06/10_zscore.py | A simple z-score of price | MCX}}

In this run gold's latest z-score is about -2.08 - meaning the price is stretched unusually *low* versus its recent average. A trader of **mean reversion** (the idea that stretched prices tend to snap back) would read a large negative z-score as a possible bounce setup, and a large positive one as possibly overextended. We'll turn z-scores into real trading bands in the pair-trading chapter.

:::key
The z-score is the same maths as the boolean mask and the volatility example combined: subtract the mean, divide by std, and you've put any price series onto a universal "how unusual is this?" scale. Because it's vectorised, you get a z-score for *every* day in one line - and counting the `|z| > 2` days is just one more mask.
:::

## Recap

- An **array** (`ndarray`) is a row of numbers built for math; get one from a price column with `.to_numpy()`.
- **Vectorisation** applies an operation to a whole array at once - no loops - and **broadcasting** stretches a single number across every element.
- **Simple returns** `(p1-p0)/p0` are intuitive; **log returns** `ln(p1/p0)` add up cleanly across time and feed volatility.
- `np.cumprod(1 + returns)` compounds daily moves into a **cumulative** growth curve.
- **Volatility** is the standard deviation (`std`) of returns, **annualised** by multiplying by `√252`.
- **Boolean masks** ask a True/False question of every element, then filter - the seed of every signal; `argmax`/`argmin` find *where* an extreme happened.
- **Correlation** (`np.corrcoef`) scores how two return streams move together; a **z-score** measures how stretched a price is from its mean.

You now have the fast-math foundation that everything else stands on. Next we step up to pandas - the library that wraps these arrays in labelled tables with dates, rolling windows and group-bys, making real market analysis effortless.
